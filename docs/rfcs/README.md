# Requests for comments

Use an RFC for changes that alter public MCP contracts, trust boundaries, persistence formats,
routing semantics, or provider integrations.

An RFC should contain:

1. problem and user impact
2. goals and non-goals
3. proposed interfaces and data flow
4. security and provider-policy implications
5. migration and compatibility plan
6. alternatives considered
7. test and rollout plan

Name files `NNNN-short-title.md`. Open an issue before submitting an RFC pull request.
